# 04 — Real-corpus replay

The only gate with a **heavy full path**. Lite is always-on and fast; full runs
a real Lake build.

## Claim

> Real-corpus replay of **`Lean4Axiomatic.Integer.one_mul_one_eqv_one`** against
> a real Lake build of the upstream repo. The always-on unit surface asserts:
> forbidden-token gating (an `axiom`/`sorry` in a proof block is rejected, while
> a repo merely *named* `lean4-axiomatic` is fine), replay-receipt determinism,
> and the pinned case metadata (`dataset_url =
> https://github.com/cruhland/lean4-axiomatic.git`, `toolchain =
> leanprover/lean4:nightly-2024-06-08`). The **full** path additionally drives
> the real Lake build of the target theorem.

## Run

```bash
# Lite (default): always-on unit tests, < 1s, no heavy deps.
bash reproduce/04_real_corpus_replay/run.sh

# Full: real Lake build (heavy).
RUN_FULL_REPLAY=1 bash reproduce/04_real_corpus_replay/run.sh
```

- **Lite runtime:** < 1s. **Lite tests:** 18 (7 `@slow` deselected).
- **Lite dependencies:** core only — the 18 unit tests do **not** import
  `pyarrow` or touch the parquet.
- **Full dependencies:** `pyarrow` + the 43MB parquet (fetch below), a
  checked-out source repo, and `elan`/`lake` @
  `leanprover/lean4:nightly-2024-06-08`.

### Fetch the parquet (full path only)

```bash
MANIFOLD_DATA_DIR=/path/to/data bash reproduce/04_real_corpus_replay/fetch_data.sh
```

Downloads `lean-github.parquet` and verifies
SHA256 `849076288d96d06f68deb5ebcbf65aefba8939fc31b6db897225ea8df26133cb`.
The parquet is **not committed** (no Git LFS — quota).

> The replay module reads `MANIFOLD_DATA_DIR` by default. Running
> `fetch_data.sh` places the parquet where the gate expects it.

## Receipt

- `receipts/canonical.json` — committed, paper-cited lite canonical (`status=FROZEN`).
  It freezes the fast claim surface: provenance checks, forbidden-token audit,
  proof-block matching, candidate-row validation, and receipt determinism.
- `receipts/canonical_full.json` — committed full canonical (`status=FROZEN`).
  It freezes the real Lake build path: `RUN_FULL_REPLAY=1` adds the seven slow
  tests, compiles the pinned source repository with the pinned Lean toolchain,
  accepts the recorded proof, and rejects the `rfl`-only falsifier.
- `receipts/latest.json` — regenerated by every run; `provenance.mode` is `lite`
  or `full`.

## What to look at

- Lite: `18 passed, 7 deselected`, gate `PASS`, `mode=lite`.
- Full: the 7 `@slow` tests additionally run a Lake build; missing deps surface
  as precise skips/errors there.

## Not claimed

- Lite mode does **not** run a Lake build — it asserts the always-on gating and
  metadata surface only. The Lake build is the `RUN_FULL_REPLAY=1` path.
